# Analisi del Progetto TS-API-LINUX

## Panoramica

**TS-API Portal** è un'applicazione web Next.js 15 che funge da intermediario tra un gestionale aziendale (Alyante) e un'interfaccia utente moderna. Il progetto sincronizza dati dal gestionale tramite API REST e li memorizza in una cache locale SQL Server per consultazioni rapide.

**Stack Tecnologico:**
- **Framework:** Next.js 15 (App Router) con React 19
- **Linguaggio:** TypeScript 5
- **Styling:** Tailwind CSS 3
- **Database:** SQL Server (cache locale)
- **API Esterne:** Alyante Gestionale API v1 (Basic Auth)
- **Package Manager:** npm

## Componenti Principali

### 1. Client API Alyante (`lib/api.ts`)

Responsabile della comunicazione con il gestionale Alyante:
- **Autenticazione:** Basic Auth + header `Authorization-Scope`
- **Configurazione dinamica:** Supporta multi-server tramite `data/config.json` con fallback su variabili d'ambiente
- **Ricerca entità:** Utilizza endpoint `_op=search` con SearchGroupDTO
- **Paginazione:** Gestisce pageSize massimo 50 per API remote, con raccolta automatica su più pagine
- **Fallback intelligente:** Se una ricerca "estesa" fallisce, ripiega su ricerca "safe" con filtraggio locale
- **Arricchimento dati:** Fetch dettagli aggiuntivi per clienti, fornitori, articoli quando necessari
- **Risorse supportate:** clienti, fornitori, articoli, ordini, righeOrdine, destinatari

### 2. Engine Sincronizzazione (`app/api/_syncEngine.ts`)

Gestisce la sincronizzazione completa dei dati dal gestionale alla cache locale:
- **Modalità Full:** Scarica tutte le pagine di ogni risorsa
- **Modalità Incrementale:** Scarica solo i record modificati dopo l'ultima sync (con overlap di 24h)
- **Job tracking:** Stato, progresso, fasi, errori registrati per ogni job
- **Cancellazione:** Supporto annullamento job in corso
- **Resource-scoped:** Possibilità di sincronizzare singole risorse
- **TTL per risorsa:** clienti/fornitori 24h, articoli 6h, ordini/righe 15min
- **Protezione troncamento:** Errore se raggiunto maxPages con dati potenzialmente rimanenti

### 3. Store SQL Server (`app/api/_syncStoreSqlServer.ts`)

Implementazione dello storage su SQL Server:
- **Tabelle cache:** `cache_clienti`, `cache_fornitori`, `cache_articoli`, `cache_ordini`, `cache_righe_ordine`
- **Tabelle metadata:** `sync_jobs`, `sync_meta`, `sync_resource_meta`
- **Indici covering:** Ottimizzati per query rapide su documenti (355ms vs 30-90s)
- **Colonne INT:** `cli_for_fatt`, `cli_for_dest` per filtraggio efficiente

### 4. Explorer UI (`app/page.tsx`)

Interfaccia principale a 3 pannelli:
- **Sidebar sinistra:** Navigazione risorse (clienti, fornitori, articoli, ordini) con conteggi
- **Pannello centrale:** Albero gerarchico espandibile con ricerca client-side
- **Pannello destro:** Dettagli del nodo selezionato

**Caratteristiche UI:**
- Navigazione gerarchica: Cliente → Documenti → Fatture/Ordini/DDT
- Caricamento on-demand dei documenti per ogni cliente/fornitore
- Classificazione automatica documenti (fatture, ordini, DDT, altri)
- Link a pagine simulate (fattura, ordine, DDT)
- Design moderno con Tailwind CSS, gradienti, backdrop-blur

### 5. Logica Albero (`lib/explorerTree.ts`)

Costruzione e manipolazione dell'albero di navigazione:
- **buildExplorerTree:** Crea struttura gerarchica dai dati grezzi
- **groupDocumentsByClass:** Classifica documenti per tipo
- **classifyDocumentType:** Riconosce tipo documento da codici testuali/numerici
- **buildNodeDetails:** Estrae campi rilevanti per il pannello dettagli
- **filterExplorerTree:** Filtro client-side sull'albero

## Endpoint API

### Locali (GET - Read-only dalla cache)
| Endpoint | Descrizione |
|----------|-------------|
| `GET /api/local/clienti` | Query clienti da SQL |
| `GET /api/local/fornitori` | Query fornitori da SQL |
| `GET /api/local/articoli` | Query articoli da SQL |
| `GET /api/local/ordini` | Query documenti da SQL |
| `GET /api/local/righeOrdine` | Query righe da SQL |
| `GET /api/local/meta` | Metadati sync |

### Sincronizzazione
| Endpoint | Metodo | Descrizione |
|----------|--------|-------------|
| `POST /api/sync/start` | POST | Avvia job sync |
| `GET /api/sync/status/{jobId}` | GET | Stato job |
| `GET /api/sync/history` | GET | Storico jobs |
| `POST /api/sync/cancel` | POST | Annulla job |

### Configurazione
| Endpoint | Descrizione |
|----------|-------------|
| `GET/POST /api/config` | Lettura/scrittura config |
| `POST /api/config/clear` | Pulizia configurazione |
| `POST /api/config/test` | Test connessione |

## Punti di Forza

1. **Architettura pulita:** Separazione chiara tra client API, sync engine, store, UI
2. **Resilienza:** Fallback multipli per ricerche API, gestione errori robusta
3. **Performance:** Cache SQL Server con indici ottimizzati
4. **UX moderna:** Interfaccia Explorer con navigazione gerarchica intuitiva
5. **Configurazione dinamica:** Supporto multi-server senza riavvio
6. **Sync incrementale:** Riduce tempo di sincronizzazione per aggiornamenti

## Aree di Miglioramento

1. **Test automatici:** Assenza di suite di test (solo mockup separato con Vitest)
2. **Type safety:** Alcuni `any` e `unknown` potrebbero essere tipizzati meglio
3. **Documentazione API:** Swagger presente ma non integrato nell'app
4. **Sicurezza:** Credenziali in `.env.local`, nessuna protezione per endpoint write
5. **Error handling UI:** Messaggi errore potrebbero essere più user-friendly
6. **Monitoraggio:** Nessun logging strutturato o metriche
